# Methods Cookbook — reusable patterns for the SoD remix

Living document. Every pattern here was implemented and verified at least once in this
project; cite the component that proved it. Add new patterns as chapter passes invent them.

## 1. Compose-don't-overwrite field patching (proved: component 100, rest 5×)
Read the CURRENT value of a binary field and map it through a transformation
(`$array(~%val%~)` lookup generated by a Python script into a `.tpa` include), instead of
writing absolute values. Composes with whatever another mod set; idempotent-ish at the
floor value. Guard with the engine's own validity conditions (for rest headers:
`count>0 && enabled && max>0`) and exclude sentinel values (100 = BDNOREST "cannot rest").

## 2. EXTEND_TOP skip-guard (proved: component 110, recruiter sites)
When a vanilla script block must not fire in a new situation and it already has a
one-shot guard variable, don't touch the block: `EXTEND_TOP` a new block that detects the
situation (`InPartyAllowDead(x)`) and pre-sets the guard + `Continue()`. Zero edits to
Beamdog/mod text, composes with anything, and the vanilla block still serves its other
purposes (e.g. rescuing a dismissed companion's world-copy).

## 3. Global pre-set to disarm a sequence (proved: component 130, dream skip)
When a whole scripted sequence is driven by a progression global with NO external
consumers (verify exhaustively first!), pre-set the global past its terminal value from
the campaign master script (`BDBALDUR.BCS` for SoD — correct on both EET and standalone;
CAMPAIGN.2DA column names the master script per campaign). Never patch shared engine
scripts like `PLAYER1D.BCS` that EET generates and other mods extend.

## 4. Surgical block neutralization via action deletion (proved: component 110, BD0103)
When independent per-subject blocks must die and have no guard variable, DECOMPILE_AND_PATCH
+ `REPLACE_TEXTUALLY` exactly the *action lines* (`ActionOverride("x",LeaveParty())`), not
whole blocks. The emptied block fires harmlessly and dies with its gate. Patterns must not
depend on decompiler comments; case-insensitive matching covers name-case variants.
Loop subjects with `PATCH_FOR_EACH`.

## 5. Cutscene skip at the launch site (proved: component 120, bdcut11)
To remove a whole cutscene, don't gut the cutscene script — replace its
`StartCutSceneEx("x",TRUE)` at the LAUNCHING script with `FadeFromColor + EndCutSceneMode()`,
after confirming all party moves/XP/plot advances happen before the launch line. The skipped
cutscene and its CUTSKIP response become unreachable dead code, which is harmless.

## 6. In-scene actor removal (proved: component 120, BD0103 + BDCUT28)
To remove one actor from a scene that must survive: delete only their `CreateCreature` line;
verify downstream logic is gated on `!See("actor")`/absence-tolerant (`ActionOverride` on a
missing object no-ops). If the actor's dialog was what ended cutscene mode, replace their
`StartDialogNoSet` with an explicit `EndCutSceneMode()` — the engine pattern is that the
dialog implicitly terminates the cutscene, and removing it without the replacement soft-locks.

## 7. False()-gating (proved: components 120 — BD5100 blocks, BDSCRY/BDIMOEN replies)
- Script block: `REPLACE_TEXTUALLY` its most distinctive trigger line with
  `False()` + newline + the original line. Whole block dead, no structure change.
  Prefer this over deleting blocks that contain `SetCutSceneLite`/state changes.
- Dialog reply: `.d` file with `ADD_TRANS_TRIGGER DLGNAME state ~False()~ DO transIdx`
  (appends/ANDs — never overwrites existing triggers, so mod-added transitions in the same
  state are safe). Verify transition indices by decompiling the LIVE dlg first.

## 8. Verification loop (proved: whole Wave 1)
After every install on the dev copy: (a) binary read-back of patched fields via Python
against the design table; (b) `weidu --out scratchpad` decompile of every patched
`.bcs`/`.dlg` and grep for: removed text absent, added text present, preserved landmarks
intact (e.g. mod-NPC blocks, scene TextScreens). "SUCCESSFULLY INSTALLED" alone proves
nothing.

## 9. Install-position + platform doctrine
Tail-install after EET_end (all patches are in-place edits of final files). Detect platform
by content, not GAME_IS alone: `FILE_EXISTS_IN_GAME ~bd0120.are~` = SoD content present;
campaign-master script names come from CAMPAIGN.2DA. Mod-NPC extension blocks inside
Beamdog scripts (Aura/Bristlelick in BD0103, Ajantis in K#TELBGT) are third-party property:
never delete or rewrite them.

## 10. Beamdog debug recipes as verified skip-spines (found: prologue pass)
Before designing a skip, search `BDDEBUG.baf` (and `bddebug`-adjacent scripts) for
Beamdog's own dev shortcut for that segment — it is the developer-tested minimal state
vector. Example: the dungeon skip is literally `bd_plot=50 + bd_npc_camp_chapter=1 +
StartCutSceneEx("bdcut00z")`. Build production skips on that spine, then add the polish
state (journals, default outcomes) research says later content reads.

## 11. Instant-exit area skip (designed: prologue pass, dungeon removal)
To remove a playable area WITHOUT changing start areas or teleport targets: let the party
land there invisibly (no control), let 2-3 area-script passes run so every bootstrap block
(vanilla, EET K#, mod-added) executes untouched, then launch the area's own vanilla exit
cutscene. Patch only the staging/walk-in blocks between bootstrap and gameplay. One
implementation covers all entry flows; never edit `SODSTRTA/CAMPAIGN.2DA` start areas —
that silently skips every script other mods patched.

## 12. Item re-homing channels (designed: prologue pass)
When cutting content that holds unique/quest items, first build a verified inventory
(containers + CRE inventories + script-granted, cross-referenced install-wide for other
copies), classify P1 quest-critical / P2 unique / generic, then deliver via two channels:
(a) boss/fight loot for thematically-owned gear, (b) an "evidence chest" with narrative
cover for the rest. Re-home flag *setters* that lived on the looting event (e.g.
`BD_SAREVOK_SECRET`), not just the items.

## 13. SCS-grade encounters, detect-and-adapt (designed: prologue pass; see 10d)
Recipe per enemy: override slot `dw#gpsht` (humanoid) / `dw#gpshm` (monster); default slot
= SCS brain — generic `dw1mel*/dw1ran*/dw2*` for non-casters (add `dw#ptn*` potions to arm
their potion blocks), a matching generated `dw#mgNNN/dw#prNNN` for casters (every cast is
`HaveSpell()`-guarded → reads the CRE's real book; prebuffs live inside the brain). Guard
each assignment with `FILE_EXISTS_IN_GAME ~dw#....bcs~`; fall back to the SoD-native stack
(`BDENSHTV` + bespoke brain) so the fight works without SCS. `dw#NNN` numbering is
install-generated: hardcode only for this install, resolve at install time if distributing.
Scale composition with explicit `Difficulty()` branches (engine slider never buffs placed
creatures; Insane only doubles enemy damage). Cheap upgrade: give bosses the consumables
their vanilla AI already references — often shipped itemless.

## 14. WeiDU tra/variable landmines (proved the hard way: prologue build, v0.2.0)
- `%MOD_FOLDER%` is NOT the mod's folder on WeiDU 24600 — it resolved to
  `weidu_external` at install time. Big mods (SCS) define it themselves before use.
  In this mod: always write literal `chriz-sod-remix/...` paths in tpa bodies.
- `COMPILE`/`EXTEND_TOP` of `.d`/`.baf` files take their `@ref` strings from
  **AUTO_TRA** (looks up `languages/<lang>/<source-basename>.tra`) or an explicit
  `USING` clause — NEVER from tp2-level `LOAD_TRA`. With our `AUTO_TRA` header, every
  compiled source that uses `@refs` needs a matching `languages/english/<basename>.tra`
  (a copy of the component tra is fine). `LOAD_TRA` still needed for @refs used in
  tpa code itself (SAY, RESOLVE_STR_REF).
- WeiDU decompiles (`weidu file.bcs`) land in the CURRENT WORKING DIRECTORY and
  ignore `--out` for positional args — run with cwd in a scratch dir, never the game root.
- **ALWAYS fully close the game (Baldur.exe + InfinityLoader.exe) before installing.**
  A running game holds `lang/en_US/dialog.tlk` open. WeiDU writes the override files
  (`.dlg` referencing new strrefs) as it goes, then flushes the TLK strings at the very
  END — a locked TLK makes that flush die with `Sys_error("...dialog.tlk: Permission
  denied")` AFTER every component already printed "Installed". Result: dialogs point at
  strrefs the TLK never got → in-game "Invalid: NNNNNN". Symptom to recognize instantly:
  `dialog.tlk` mtime older than the install; entry count unchanged. Fix = close the game,
  then add a new tail recovery component that safely reapplies the affected dialog/string
  patches; never force-install or reinstall an existing WeiDU.log entry. Verify by
  decompiling one repaired `.dlg` and confirming the SAY strref resolves to real text.

## 15. Live-save semantics (recurring caveat)
`.are` rest-header/actor changes: only areas NOT yet visited on that save. `.bcs`/`.dlg`:
next area (re)load; scenes whose gate global already advanced never re-fire (usually the
desired behavior). `.cre`: newly spawned instances only; joined companions are baked into
the save.
